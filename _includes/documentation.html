About

Chronix natively speaks time series and its design allows you to store nearly every kind of data type within a time series. It let you decide how a time series look like. Chronix Kassiopeia is a specialized time series package that can easily plugged into Chronix. It also delivers some high-level time series functions, e.g., trend detection, frequencies, outliers. But you can decide if you want to use Kassiopeia or your custom time series within Chronix. All you have to do, is to implement a converter with two methods.

Chronix is built to store time series highly compressed and for fast access times. On benchmarks Chronix not only takes 5 to 171 times less space than related time series databases to store time series data, but also saves up to 83% of the access time, and up to 78% of the runtime on a mix of real world queries. For the measurements we used a commodity hardware laptop computer and Chronix using the Apache Solr scenario (single node).

Chronix comes with three different scenarios with different goals:

  1. Chronix Storage (based on Apache Lucene)
  2. Chronix Server (based on Apache Solr)
  3. Chronix Cluster (based on Apache Spark)

You can use Chronix as a small storage library (Scenario 1) and plug it into your application. It then stores the time series using Apache Lucene. Or, if you need / prefer a typical client-server scenario use Chronix with Apache Solr (Scenario 2). Apache Solr brings several useful features like scalability, fault tolerance, distributed indexing, or replication. When you need a parallel and distributed time series processing then use Scenario 3 with Spark. Time series are stored in a Chronix Storage and HFDS or in a Chronix Server cluster. The integration of Chronix into Spark allows a parallel time series processing.
